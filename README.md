# InternRadar AI

> A verification-first internship tracker for undergraduate CS students, focused on official application links, freshness checks, and AI-assisted fit tags.

**Live demo:** https://internradar-ai.vercel.app/

## Problem

Internship hunting for undergraduates is noisy and unreliable. Aggregators and
spreadsheets fill up with dead links, stale postings, and roles that quietly
require U.S. citizenship or won't sponsor international students. Freshmen and
sophomores waste hours on listings that were never open to them in the first
place, and AI "match" features often invent qualifications that aren't in the
posting.

InternRadar AI exists to fix the trust problem first: every entry points to an
**official application link**, carries a **last verified date**, and states a
clear **open / closed / uncertain** status before anything else.

## What Makes This Different

This is **not** a generic copy of large internship lists. The emphasis is on
quality and verifiability, not volume:

- **Official application links** — entries link to the company's own posting
  (career page, Greenhouse, Lever, Ashby, Workday), not a re-hosted copy.
- **Last verified date** — you always know how fresh the information is.
- **Open / Closed / Unclear status** — honest status, never an optimistic guess.
- **Work authorization notes** — captured when stated by the posting.
- **U.S. citizenship requirement detection** — flagged when the posting says so.
- **Sponsorship / CPT / OPT ambiguity notes** — ambiguity is recorded as
  ambiguity, not resolved by guessing.
- **Freshman / sophomore / undergraduate friendliness** — surfaced when the
  posting gives evidence.
- **Evidence-based AI fit tags** — AI relevance and fit summaries are derived
  only from text in the posting, never hallucinated.
- **No copied full job descriptions** — we store links and short factual notes,
  not pasted JD text.

## Current MVP Scope

This phase is **local scaffolding only**:

- Repository structure, documentation, and contribution rules
- A JSON schema for internship entries (`data/schema.json`)
- A validated data file (`data/internships.json`)
- Standard-library-only tooling for validation, auditing, and doc generation
- Search, verification, and data-quality policies

The scaffolding came first and intentionally started empty; the dataset now holds
a small batch of manually verified postings (see **Status** below).

## Data Quality Principles

1. **No hallucinated jobs.** Every entry corresponds to a real, official posting.
2. **No copied full job descriptions.** Store links and short factual notes only.
3. **Verify, don't guess.** When a fact (status, citizenship, sponsorship) is
   unclear, it is recorded as `Unclear`.
4. **Freshness is a feature.** Every entry carries a `last_verified_date`.
5. **Evidence over inference.** Claims about citizenship, sponsorship, or fit
   must be backed by `evidence_notes`.

## Planned Fields

Each internship entry is defined by `data/schema.json` and documented in
[`docs/data_fields.md`](docs/data_fields.md). Highlights include:

- `company`, `role`, `category`, `location`, `location_type`, `internship_term`
- `application_url`, `source_url`, `source_type`
- `status`, `last_verified_date`, `discovered_date`, `age_days`
- `tech_keywords`, `ai_relevance`, `full_stack_relevance`
- `student_level`, `freshman_sophomore_friendly`
- `requires_us_citizenship`, `sponsorship_note`, `work_authorization_note`
- `evidence_notes`, `fit_summary`, `risk_flags`
- `date_added`, `date_updated`

## Local Workflow

Tooling lives in `scripts/` and is **standard-library Python only** (no installs).
The future workflow for adding a verified posting is:

1. **Create a pending draft** — run `python scripts/new_submission.py` and answer
   the prompts. It writes a Markdown draft to `pending/` (named
   `YYYY-MM-DD_company_role.md`). It refuses to create a draft without an
   `application_url` and a `last_verified_date`.
2. **Review the official application page manually** — confirm the role, status,
   eligibility, and any citizenship/sponsorship language. Fill in the remaining
   fields in the draft, quoting only short evidence snippets (never full JDs).
3. **Convert the reviewed Markdown to JSON** — run
   `python scripts/submission_to_json.py pending/<file>.md`. It validates enum
   values, generates an `id`, computes `age_days`, and prints a JSON object to
   stdout.
4. **Save that JSON object as a temporary local file** — e.g. redirect the output
   to a scratch file you keep outside the dataset.
5. **Append it** — run `python scripts/append_entry.py <json-file>`. It refuses
   duplicate `id`s and entries missing `application_url` or `last_verified_date`,
   then appends and re-sorts `data/internships.json`.
6. **Validate** — run `python scripts/validate_data.py` and resolve any errors or
   warnings.

A human-fillable form lives at
[`templates/internship_submission_template.md`](templates/internship_submission_template.md).
The `pending/` and `reviewed/` folders hold drafts before and after review.

> Entries appear only after a real posting is manually reviewed against its
> official page and added through the steps above. The dataset currently holds a
> small batch of verified postings (see **Status** below).

## Maintainer Commands

All scripts are **standard-library Python only** (no installs). After the data
changes, regenerate the derived docs and re-validate:

```
python scripts/validate_data.py          # schema/field validation
python scripts/audit_data_quality.py     # local-only quality audit (links, evidence, flags)
python scripts/generate_readme_table.py  # refresh README table + docs/internships_table.md
python scripts/generate_status_report.py # refresh docs/status_report.md
```

The full maintainer process — including when to mark a role `Closed`, when to
update `last_verified_date`, and re-verification of old postings — is documented
in [`docs/maintenance_workflow.md`](docs/maintenance_workflow.md).

## Architecture

One trusted JSON dataset, a layer of standard-library Python tools that validate
and derive artifacts from it, and a read-only Next.js dashboard on top — no
database, no backend. `data/internships.json` is the single source of truth;
everything else (README table, status report, the dashboard's data copy) is
generated from it. See [`docs/architecture.md`](docs/architecture.md).

## Quality Gate

Run the whole local pipeline with one command:

```
python scripts/check_all.py
```

It runs validation → quality audit → README-table generation → status-report
generation → web-data sync → web build (if npm is available), stopping on the
first failure. It only writes generated artifacts; it never edits the source
dataset directly.

## Automated Candidate Discovery

InternRadar AI can **discover** internship candidates from public sources and
**verify** their official application pages, producing reviewable drafts in
`pending/auto/` and a report at
[`docs/candidate_review_report.md`](docs/candidate_review_report.md). This is
**assisted automation, not blind auto-append** — it **does not** add jobs to the
verified dataset. Promotion into `data/internships.json` is always an explicit,
human-reviewed step.

```
python scripts/discover_candidates.py                          # public sources -> tmp/candidates_filtered.json
python scripts/verify_candidates.py --limit 10                 # verify pages -> pending/auto/*.md + report
python scripts/promote_candidate.py pending/auto/<file>.md     # explicit, after you review the official page
python scripts/check_all.py                                    # validate + regenerate + build
```

Drafts marked **"Needs manual review"** require `--allow-needs-review` to promote.
See [`docs/automation_policy.md`](docs/automation_policy.md) for allowed sources,
verification rules, and the confidence scoring.

### Full-auto update

`scripts/auto_update_verified.py` runs the whole pipeline in one command and
**conservatively auto-promotes only high-confidence verified postings**
(`verification_confidence >= 90` and no hard blocker). Lower-confidence candidates
stay in `pending/auto/` for human review; nothing from a private/login-gated board
or a non-official link is ever promoted, and compensation/work-authorization are
never guessed (`Unclear` when the official page is silent).

```
# Dry run (default — no dataset changes, no commit, no push):
python scripts/auto_update_verified.py --limit 20

# Full auto (apply, commit, and push):
python scripts/auto_update_verified.py --limit 50 --max-promote 5 --min-confidence 90 --apply --commit --push
```

`--push` requires `--commit`, and `--commit` requires `--apply`. The run refuses
to commit if validation, audit, build, or any data invariant fails.

Because the tracker targets undergraduates, **graduate-only, PhD-only, MBA-only,
and advanced-degree-only roles are blocked from auto-promotion** (mixed-eligibility
roles such as "Bachelor's or Master's" are not). Hardware-only and non-technical
roles are likewise excluded. **Auto-promotion is limited to clearly
undergraduate-appropriate software/AI/data internships; hardware-adjacent,
graduate-only, and ambiguous program roles (titles without a clear internship/co-op
designation) stay in manual review.**

Generated pending drafts (`pending/auto/auto_*.md`) are local review artifacts and
are ignored by Git; the verified dataset remains `data/internships.json` and the
tracked review summary is [`docs/candidate_review_report.md`](docs/candidate_review_report.md).

### Re-verify and prune closed postings

`scripts/reverify_existing.py` re-checks existing postings against their official
pages and **conservatively removes** ones that have closed. A posting is removed
**only** on a deterministic failure (HTTP 404/410, an explicit closed/filled/
expired banner, a final URL that is now generic/private/api/search, or the role
title gone with no apply flow). **Transient errors — timeout, DNS, HTTP 429, HTTP
5xx — and ambiguity never remove a posting.** Every removal is archived to
[`archive/removed_internships.json`](archive/removed_internships.json) before
deletion, and a per-run `--max-remove` cap (default 5) bounds the blast radius.

```
# Dry-run (default — no data/archive changes):
python scripts/reverify_existing.py

# Apply (prune deterministic failures, archived first; cap 5):
python scripts/reverify_existing.py --apply --max-remove 5

# Prune closed AND add new postings in one pipeline run:
python scripts/auto_update_verified.py --limit 50 --max-promote 5 --min-confidence 90 --apply --prune-closed --max-remove 5
```

### Daily schedule

The full-auto pipeline runs **daily on a GitHub Actions schedule** (and can still
be run manually) via the **"Auto Update Verified Internships"** workflow — adding
new high-confidence postings and pruning deterministically-closed ones (dry-run by
default for manual runs; see [`docs/automation_policy.md`](docs/automation_policy.md)).
The cron is UTC (`0 15 * * *` ≈ 08:00 America/Los_Angeles, ±1h across DST).

## Dashboard MVP

A local **Next.js** dashboard now lives in [`web/`](web/) — a searchable,
filterable, read-only view of the verified listings (Next.js App Router +
TypeScript + Tailwind CSS, no backend or database).

- The **source of truth remains `data/internships.json`**. The dashboard reads a
  generated copy at `web/src/data/internships.json`.
- Sync the data into the dashboard with:
  ```
  python scripts/sync_web_data.py
  ```
- Run it locally from inside `web/` with `npm install` then `npm run dev`.
- The dashboard is a **frontend view only** — it never applies to jobs, sends
  emails, or edits the dataset. See [`web/README.md`](web/README.md).

## Deployment

The dashboard source lives in [`web/`](web/) and can be deployed on **Vercel**
straight from this GitHub repo:

- **Root Directory:** `web` (the Next.js app is not at the repo root)
- **Framework Preset:** Next.js · **Build:** `npm run build` · **Install:** `npm install`
- **Environment variables:** none required for the current MVP
- Run `python scripts/check_all.py` before deploying, and if
  `data/internships.json` changed, re-run `python scripts/sync_web_data.py` and
  commit the updated `web/src/data/internships.json` first (Vercel builds the
  committed copy and does not run the Python sync).
- **Live demo:** https://internradar-ai.vercel.app/

Full step-by-step settings are in
[`docs/deployment_notes.md`](docs/deployment_notes.md).

## Disclaimer

InternRadar AI is an aid, not an authority. Postings change without notice.
**Always verify every detail — status, deadlines, eligibility, citizenship and
sponsorship requirements — on the company's official application page before
applying.** Nothing here is legal or immigration advice.
